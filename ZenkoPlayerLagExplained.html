The Real 3-Phase Flow

Your code does not simply "block everything." It runs a precise 3-phase cycle. Understanding this order is everything.

1
Auto-Detection
ServerDetect script finds game server IP:PORT → iptables rule injected → traffic routed to NFQUEUE
2
FLUSH DROP
First flush_drop_count packets are immediately dropped (not held). This creates initial desync.
3
HOLD QUEUE
Every subsequent packet to current_port is intercepted and stored in a linked list. No verdict is sent yet.
4
RLS RELEASE
On Unblock, held packets are processed front-to-back through your rls_pattern. Accept = send. Drop = discard.

Critical: The RLS "drop" action drops already-held packets during release. It does not drop new live traffic. Once unblock finishes, normal traffic resumes instantly.

RLS Pattern Mechanics

The pattern is an array of instructions read sequentially from config.json. When unblock fires, the engine walks your held-packet queue and applies each rule.

How It Reads (Step by Step)

  1. Unblock triggered → all held packets become an ordered array.
  2. Read Pattern[0]: if action:"drop" count:15 → first 15 held packets are discarded.
  3. Read Pattern[1]: if action:"accept" count:17 → next 17 held packets are sent to server.
  4. Continue until array ends or packets run out.
  5. If packets remain after last pattern → rls_pattern_remainder decides their fate (keep this as "accept").
config.json — RLS Section
// Maximum 16 entries. Remainder should ALWAYS be "accept" "rls_pattern": [ { "action": "drop", "count": 15 }, { "action": "accept", "count": 17 }, { "action": "drop", "count": 8 }, { "action": "accept", "count": 12 } ], "rls_pattern_remainder": "accept"

How many packets are immediately dropped when Block turns ON, before holding begins. Creates the initial "cut" in sync.
Affects Phase 2
rls_pattern[]
The release schedule. Each entry is {action, count}. Max 16 entries. Controls which held packets survive unblock.
Affects Phase 4

Fine Tuning for Newbies

Tuning is the art of making the server believe your delayed packets are just bad mobile internet, not a cheat. Here is how to adjust each parameter safely.

1. Tuning flush_drop_count

This is your opening cut. It drops live packets instantly when Block turns on. If too high, the server loses too much sync immediately and backtracks you. If too low, the effect is weak.

flush_drop_count10
Effect: Moderate initial desync. Safe for most tickrates.
  • 0-5: Subtle. Server barely notices. Good for testing.
  • 8-15: Sweet spot. Creates a clean break without instant rubberband.
  • 20+: Aggressive. High risk of timeout or heavy correction.

2. Tuning rls_pattern

This is the main event. It decides the fate of every held packet during unblock. The pattern runs front-to-back on your queue.

Safe Starter:
drop 8 → accept 12
Gentle. Good for high-tickrate games.
Balanced:
drop 12 → accept 15 → drop 6 → accept 10
Varied. Mimics real network jitter.
Aggressive:
drop 20 → accept 10
Heavy desync. Risky on strict anti-cheat.

Golden rule: Your total held packet count must be roughly flush_drop + (sum of all pattern counts). If you hold 50 packets but your pattern only covers 30, the remainder handles the rest. That is why remainder = "accept" is critical — it acts as a safety net.

3. The "Human Lag" Rule

Real mobile lag is irregular. Never use the same drop/accept numbers forever. Alternate them so the server sees a believable jitter graph.

Good vs Bad Pattern Design
// BAD — robotic, easy to fingerprint "rls_pattern": [ { "drop": 15 }, { "accept": 15 }, { "drop": 15 }, { "accept": 15 } ] // GOOD — mimics real network breathing "rls_pattern": [ { "drop": 8 }, { "accept": 12 }, { "drop": 14 }, { "accept": 10 }, { "drop": 6 }, { "accept": 16 }, { "drop": 10 }, { "accept": 11 } ]

4. How Long Should You Hold?

There is no "hold duration" timer in your code. Hold time depends entirely on how long you keep Block ON. If the game sends 60 packets/sec and you block for 2 seconds, you hold ~120 packets. Your RLS pattern must be sized accordingly.

  • Short block (0.5s): ~30 packets held. Use small pattern counts (5-10).
  • Medium block (1-2s): ~60-120 packets. Use medium counts (10-20).
  • Long block (3s+): 180+ packets. You need longer patterns or the remainder will do all the work.

Why This Avoids Backtrack

Normal packet blocking tells the server: "Player was at A, then nothing, then suddenly Z." The server says impossible and snaps you back to A.

Your hold+RLS system tells the server: "Player was at A, then B, C, D, E arrived late because of lag." The server sees a complete history, just delayed. It blames your internet, not your movement.

Simple Block (Backtrack)

  • Packet 1-20: Dropped forever
  • Packet 21: Arrives with huge position jump
  • Server detects teleport
  • Instant snapback to old position

Hold + RLS (No Backtrack)

  • Packet 1-20: Held in memory
  • Unblock → replayed in order via pattern
  • Server receives A→B→C→D→E
  • Interprets as network lag spike

Newbie Mistakes That Break It

Remainder = "drop"

If your pattern runs out and remainder is drop, all leftover held packets are discarded. The server sees a massive sync hole → instant backtrack or timeout.

Pattern Too Short

If you hold 100 packets but your pattern only covers 30, the remainder handles 70. If remainder is accept, you dump 70 packets at once → burst detection.

flush_drop Too High

Dropping 30+ packets before holding begins creates an instant desync chasm. The server may reconcile before you even release anything.

Holding Too Long

Blocking for 5+ seconds queues 300+ packets. Releasing them all through a tiny pattern takes forever. Server may disconnect due to stale state.

Static Patterns

Using the exact same drop/accept count repeatedly looks like a robot. Anti-cheat heuristics can fingerprint perfect repetition.

Forgetting Order

Your code preserves order automatically, but if you ever modify it to reorder packets, the sequence ID jump will flag instantly. Never shuffle.

Recommended Tuning Workflow

1
Start Conservative
flush_drop_count: 5 | pattern: drop 5 → accept 10
2
Test Movement
Walk, sprint, jump. Watch for snapback or freezing.
3
Increase Gradually
Bump flush_drop to 8. Add pattern steps. Never double everything at once.
4
Watch Bad Signs
Freeze, snapback, screen shake, timeout → reduce drop counts immediately.
5
Add Variation
Mix counts: 8/12, 12/15, 6/10. Real lag is never uniform.

Good Signs vs Bad Signs

Good Signs

  • Movement feels delayed but smooth
  • No instant teleport back
  • Server accepts new position
  • Connection stays alive
  • Damage/hits register normally

Bad Signs

  • Character freezes in place
  • Instant snapback to old spot
  • Screen shakes violently
  • Connection timeout / disconnect
  • Other players see you teleport

The Mailbox Analogy

Instead of burning letters,
you collect them in a box,
then mail them in batches later.

The server thinks: "Slow post office"
Not: "This person teleported"

flush_drop = the first letters you throw away to start the trick.
rls_pattern = the schedule for mailing the rest.
remainder = accept = if you have extra letters, mail them anyway.
Never leave mail in the box forever.
